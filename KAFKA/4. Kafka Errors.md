# ==Kafka Errores==

Apache Kafka es un sistema de mensajería distribuido diseñado para manejar altos Volúmenes de datos en tiempo real. Sin embargo, como cualquier sistema distribuido, los errores ocurren y pueden afectar la entrega y el procesamiento de mensajes

En aka los errores típicos se relaciona con:

1. ## **<mark style="background:#40a9ff">Fallas temporales:</mark>**
	Problemas momentáneos como caída de un servicio externó o errores de red. Estos pueden resolverse con reintentos automáticos
	
2. ## **<mark style="background:#40a9ff">Fallas definitivas:</mark>**
	Mensajes Corruptos, inválidos o que nunca se podrán procesar correctamente. Se maneja un **DLQ (Dead Letter Queue)** para no bloquear el flujo normal.

3. ## **<mark style="background:#40a9ff">Duplicación o perdida de mensajes</mark>**
	Kafka asegura al menos una vez la entrega lo que puede generar duplicados. La perdida ocurre si no se configura correctamente el productor o consumidor

EL manejo de errores en kafka implica decidir que hacer con cada mensaje fallido, asegurando disponibilidad, consistencia y confiabilidad del sistema. Para lograrlo se utiliza herramientas y patrones como reintentos, DLQ, reprocesamiento y commits controlados

## **<mark style="background:#40a9ff">Retries / Reintentos</mark>**
Cuando el consume falla al procesar un mensaje, lo vuelve a intentar

### <mark style="background:#affad1">¿Cuando fallas las cosas?</mark>
- Timeout de la base de datos
- Servicio externo caído
- Error de Red

### <mark style="background:#affad1">Tipos de Retries:</mark>
- **Retries automaticos (misma partición)**
	- El Offset no se comitea
	- Kafka vuelve a entregar el mensaje
	- Riesgo: Bloqueo de la partición si siempre falla
-** Retries con Delay**
Kafka no tiene delay nativo si se hace un:
	- Retry topics (**```topic.retry.1s,...5s,...etc```**)
	- Mensaje se replica con backoff
Esto evita que un mensaje "envenenado tiene toda la partición"

---
## **<mark style="background:#40a9ff">DLQ (Dead Letter Queue)</mark>**
Un topic especial donde mandas mensajes que no se pueden procesar

### <mark style="background:#affad1">¿Cuando usar DLQ?</mark>
- Error de validación
- Datos Corruptos
- Error lógico (null, formato invalido, etc)

### <mark style="background:#affad1">Flujo Tipico</mark>
- Consumes mensaje
-  Reintentas N veces
- Sigue Fallando
- Lo manda al DLQ
- Commit del Offset
Asi no se pierde el mensaje y el consumer sigue vivo

---
## **<mark style="background:#40a9ff">Reprocesamiento</mark>**
Volver a procesar un mensaje:
- Desde DLQ
- desde un offset anterior
- Vuelve a publicar en un Topic Original

### <mark style="background:#affad1">Formas Comunes</mark>
- Desde un DLQ
- Un Job lee el DLQ
- Corrije/Transforma
- Lo manda al DLQ
- Commit del Offset
Asi no se pierde el mensaje y el consumer sigue vivo

### <mark style="background:#affad1">Reset de offsets</mark>
- Kafka **```consumer.group --reset-offset```**
- Reprocesa todo o parte del topic
- Riesgo: Duplicación si su sistema no es indeponente

---
## **<mark style="background:#40a9ff">PERDIDA VS DUPLICACIÓN DE MENSAJES</mark>**

| <center>Perdida de mensajes</center> | <center>Duplicación de mensajes</center>                 |
| ------------------------------------ | -------------------------------------------------------- |
| Commit antes de Comenzar             | Consumer falla después de procesar pero antes del commit |
| ```auto.commit=true```               | Rebalance                                                |
| Producer con acks=V                  | Reintentetos                                             |
| Retención muy corta                  |                                                          |
- Kafka prefiere duplicar antes que perder
- Es normal y esperado
---
## **<mark style="background:#40a9ff">Manejo en sistemas Reales</mark>**
### <mark style="background:#affad1">Idempotencia</mark>
Procesar el mismo mensaje de una vez sin efectos secundarios
- Claves Unicas
- offsets
- Duplicación de mensajes por ID

### <mark style="background:#affad1">Commit Manual</mark>
- Procesas primero
- Commit si todo salió bien

### <mark style="background:#affad1">Exactly Once</mark>
Kafka lo soporta pero:
- Mas complejo
- Mas overhead
- No siempre necesario
